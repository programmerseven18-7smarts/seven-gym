Bagian 1

Gambaran Umum Sistem

1.1 Tujuan Project

Seven Gym Management System adalah aplikasi admin untuk mengelola operasional gym multi cabang dari pendaftaran member, check-in, kelas, personal trainer, POS, stok produk, promo, notifikasi, laporan, sampai role dan akses.

Versi saat ini adalah prototype dengan dummy data dinamis. Data dapat berubah di UI, tersimpan di localStorage untuk simulasi, dan alurnya dirancang supaya nanti bisa dipindah ke backend real tanpa mengubah pola kerja besar.

1.2 Komponen Utama

KomponenIsiFungsi
Admin DashboardNext.js, TypeScript, TailwindAntarmuka operasional owner, staff, trainer, dan portal member.
Prototype DataReact Context + localStorageSimulasi database, CRUD, invoice, queue WhatsApp, audit log, dan pengaturan.

POSKatalog + checkout modalMenjual membership, paket PT, dan produk gym dari satu layar kasir.

1.3 Prinsip Desain

  • Halaman data utama memakai tabel di desktop dan berubah menjadi card ketika mobile.
  • Form tambah/edit CRUD memakai popup responsif, bukan form panjang di bawah tabel.
  • Detail data berpindah ke halaman lanjutan, bukan panel kecil di samping tabel.
  • Menu mengikuti role dan permission, bukan sekadar disembunyikan manual.
  • Multi cabang berada di satu company Seven Gym; branch access menentukan cabang yang bisa dipakai user.

Alur Bisnis Utama

2.1 Flow Member Baru

Staff login

  -> tambah member
  -> pilih paket membership
  -> transaksi POS atau invoice
  -> pembayaran berhasil
  -> membership aktif
  -> member bisa check-in dan booking kelas/PT
1. RegistrasiData member, kontak, status, paket, dan home branch dibuat.
->
2. PembayaranKasir memilih item POS, promo, metode bayar, lalu sistem membuat transaksi dan invoice.
->
3. AktivasiMember aktif, dapat QR check-in, point loyalty, dan histori kunjungan.

2.2 Flow Operasional Harian

  1. Staff memilih cabang aktif sesuai branch access user.
  2. Member datang dan discan lewat menu Check-In.
  3. Jika membership aktif dan cabang diizinkan, member masuk ke daftar sedang gym.
  4. Booking kelas dan booking PT dibuat dari menu operasional atau dari alur kasir.
  5. Produk yang dijual melalui POS mengurangi stok cabang.
  6. Reminder WhatsApp masuk ke queue otomatis berdasarkan trigger.

2.3 Flow Laporan

Owner membuka laporan untuk melihat pendapatan, member aktif, retensi, performa trainer, kelas, produk, dan kehadiran. Data laporan mengambil ringkasan dari transaksi POS, invoice, check-in, kelas, PT, dan inventory.

4.2 Aturan CRUD

  • Detail data dibuka ke halaman lanjutan agar tabel tetap bersih.
  • Delete memakai konfirmasi dan menulis audit log.
  • Status aktif/nonaktif berpengaruh ke pilihan dropdown di alur operasional.

POS membutuhkan permission pos.view untuk melihat halaman dan pos.create atau payments.create untuk membuat transaksi. Selain itu cash shift harus aktif agar tombol bayar bisa dipakai.

Bagian 6

Operasional Gym

6.1 Check-In

  • Scan QR member melakukan validasi membership aktif, status freeze/expired, dan akses cabang.
  • Member yang sedang gym tampil di live list dan bisa checkout.
  • Histori check-in dipakai untuk laporan kehadiran dan trigger member tidak aktif.

6.2 Kelas Gym

  • Jadwal kelas menyimpan nama kelas, trainer/instruktur, jam, ruangan, kapasitas, dan status.
  • Booking kelas mencatat peserta, status booked/waitlist/attended/cancelled.
  • Waiting list bisa dipromote ketika ada peserta batal.

6.3 Personal Trainer

  • Data trainer berisi profil, cabang, spesialisasi, rate, rating, target sesi, dan komisi.
  • Booking PT mencatat member, trainer, tanggal, jam, durasi, fokus latihan, dan status sesi.
  • Trainer memiliki portal sendiri untuk jadwal, client, program, dan komisi.

6.4 Produk & Maintenance

Inventory memisahkan data produk, stok cabang, barang masuk, barang keluar, supplier, dan maintenance alat. Penjualan produk dari POS mengurangi stok, sedangkan barang masuk/keluar menjadi catatan movement.

8.2 Migrasi dari Prototype ke Real

  1. Ganti localStorage/context menjadi database PostgreSQL.
  2. Buat schema Prisma untuk company, branch, user, member, transaction, inventory, class, PT, dan notification.
  3. Pindahkan CRUD popup ke server action/API dengan permission guard.
  4. Tambahkan auth real: login, logout, token/session, refresh, dan branch selection.
  5. Integrasikan WhatsApp provider untuk queue, delivery status, retry, dan template approval.
  6. Tambahkan test untuk role access, POS checkout, stock movement, dan invoice.

8.3 Catatan Prioritas

  • POS dan payment harus paling stabil karena memengaruhi revenue dan stok.
  • Branch access harus konsisten di setiap query agar staff cabang tidak melihat data cabang lain.
  • Audit log wajib untuk action sensitif: void/refund, role change, delete data, status payment, dan close shift.
  • Export laporan bisa dimulai dari CSV/JSON, lalu ditambah PDF jika format final sudah jelas.
